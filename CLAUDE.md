# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

The personal website served at https://nahornyioleh.be (custom domain set in `CNAME`) via
**GitHub Pages, deployed by GitHub Actions** (repo Settings → Pages → Source = "GitHub Actions";
the API `build_type` is `workflow`). Pushing to `main` triggers `.github/workflows/deploy.yml`,
which builds and publishes the site. Do **not** expect "edit a file, push, it's live" — the
published site is the *assembled build artifact*, not the raw `main` tree (see Build & deploy).

The repo holds two distinct kinds of content:

1. **The landing page — source lives here.** It is a **Vite + React + TypeScript + Tailwind v4**
   project whose source is committed: `src/`, the `index.html` Vite entry, `package.json`,
   `package-lock.json`, `vite.config.ts`, `tsconfig*.json`. Change the landing page by editing
   `src/` and rebuilding — never by hand-editing built output.
2. **The sub-projects — built output, source NOT here.** Each `/{dir}/` is the *built output* of an
   external project, copied in (mostly Vite/React builds, recognizable by hashed asset names like
   `assets/index-CzAtAFVp.js`). Their source is elsewhere. Do not hand-edit those minified bundles —
   they are regenerated upstream and copied here; flag it instead of editing.

## Build & deploy

- Scripts: `npm install`, `npm run dev` (local dev server), `npm run build` (`tsc -b && vite build`
  → `dist/`), `npm run preview`. Node 22.
- `.github/workflows/deploy.yml` on push to `main` (or manual dispatch): `npm ci` → `npm run build`
  → **assemble** a `_site/` publish dir → `upload-pages-artifact` → `deploy-pages`.
- The build emits **only the landing page** (`dist/index.html` + `dist/assets/`). The assemble step
  `rsync`s the committed static content — **including every sub-project** — into `_site/` (excluding
  source/tooling), drops the stale root landing bundles, then overlays `dist/`. It **merges** the new
  landing page over the existing site; it never replaces the whole tree. Do not change the assemble
  step in a way that would drop the sub-projects.
- A directory still deploys (reachable by URL) even if it is delisted from the landing page. E.g.
  `verbes/ru` exists on disk but is no longer linked. To remove a route entirely, delete its dir.

## .gitignore

- Ignored: `node_modules/`, `dist/`, `_site/`, `dev/`, `*.tsbuildinfo`, `*.log`.
- `*.json` is ignored except allow-listed entries: **data files** (`book*.json`, `french_*.json`,
  `product_*.json`, `quiz/assets/data/*.json`) and **build config** (`package.json`,
  `package-lock.json`, `tsconfig*.json`). Adding a JSON data file that must ship → confirm it matches
  an allow-list pattern or `git add -f` it.
- `src/` and `vite.config.ts` are **tracked** (landing-page source). Note: Tailwind v4's automatic
  source detection skips gitignored paths, so `src/index.css` declares `@source "./**/*.{ts,tsx}"`
  explicitly — without it the build is silently unstyled.

## Structure

- `index.html` — the **Vite entry (source, not the served page)**: `<head>` meta/favicons,
  schema.org JSON-LD (`hasPart` lists the articles), a no-FOUC theme script, `<div id="root">`, and
  `<script src="/src/main.tsx">`. The page that ships is `dist/index.html`, generated by the build.
- `src/` — the landing-page React app: `App.tsx`, `components/` (`Header`, `ProjectCard`,
  `SystemClock`, `ThemeToggle`), `hooks/useTheme.ts` (class-based light/dark, persisted), `index.css`
  (Tailwind v4 + `@source` + `@custom-variant dark`), and **`data/projects.ts` — the list of project
  cards rendered on the page**.
- Top-level asset dirs (`icons/`, `assets/`, `images/`, `static/`, `favicon_io/`, `css-dist/`) serve
  the landing page. `dist/assets/` (hashed landing JS/CSS) is merged into `assets/` at deploy time.
- Each other directory is a self-contained sub-site reached at `/{dir}/`, each with its own
  `index.html` and (usually) its own `assets/`. They are independent — assets are scoped per project.

## When changing the site

- **Adding a sub-project:** create `/{name}/`, copy its built output in, then update
  `src/data/projects.ts` (the card list rendered on the landing page), `index.html` JSON-LD
  `hasPart` (only if it is an article), `sitemap.xml` (new `<url>` + `lastmod`), and `README.md`.
- **Removing/renaming:** update the same files together. Deleting the directory removes the route;
  leaving it delists the card while the route stays reachable.
- Use **relative paths** (`./assets/...`) in HTML — the site is served from the domain root but
  sub-projects must remain portable under their own path. (`vite.config.ts` sets `base: './'`.)
- Sub-project asset links are content-hashed; when replacing a sub-project build, update the
  `<script>`/`<link>` references in that project's `index.html` to the new hashes.
- Never hand-edit a sub-project's minified bundle or the landing's `dist/` output — edit `src/` and
  rebuild.
